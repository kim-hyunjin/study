    Kafka Docker Compose (KRaft 모드) 설정 가이드¶

    이 문서는 프로젝트에 구성된 카프카 클러스터(apache/kafka:latest)의 설정 방식과 실행 방법, 그리고 Java 클라이언트와의 연결 구조를 설명합니다.

    1. 개요¶

    작성된 설정은 KRaft(Kafka Raft) 모드를 사용합니다. 이는 기존의 Zookeeper 없이 카프카 브로커들이 스스로 클러스터 상태를 관리하는 최신 방식입니다. 3개의 브로커 노드가 하나의 클러스터로 docker-compose.yml을 통해 실행됩니다.

    공식 카프카 이미지(apache/kafka)를 사용하여, 별도의 설정 파일 없이 docker-compose.yml 내의 환경 변수(KAFKA_*)만으로 클러스터 구성 및 네트워크 설정이 가능합니다.

    2. 주요 환경 변수 설정 상세 설명¶

    docker-compose.yml에 정의된 핵심 environment 설정은 다음과 같습니다.

    A. 클러스터 및 역할 설정¶

    • CLUSTER_ID: 클러스터를 하나로 묶어주는 고유 식별자입니다. 세 개의 노드가 동일한 값을 가져야 하나의 클러스터로 묶입니다.
    • KAFKA_NODE_ID: 클러스터 내에서 브로커 자신을 식별하는 고유 ID입니다. (1, 2, 3)
    • KAFKA_PROCESS_ROLES=broker,controller: 해당 노드가 데이터 저장(broker)과 클러스터 관리(controller) 역할을 모두 수행함을 의미합니다.
    • KAFKA_CONTROLLER_QUORUM_VOTERS: 리더를 선출하고 메타데이터를 동기화할 컨트롤러들의 전체 주소 목록입니다. (예: 1@kafka-1:9093,2@...)

    B. 네트워크 및 리스너 분리 (Listeners)¶

    클러스터 내부 통신과 외부(Java 클라이언트 등) 접근을 분리하여 포트 충돌 및 통신 오류를 방지합니다.

    • INTERNAL 망 (19092 포트): 도커 네트워크 내부에서 브로커들끼리 서로 통신할 때 사용됩니다. (예: kafka-1:19092)
    • EXTERNAL 망 (9092 포트): 호스트(사용자 PC)에서 접속할 때 사용됩니다. docker-compose.yml의 ports 설정을 통해 각각 9092, 9093, 9094 포트로 매핑되며, 브로커는 클라이언트에게 자신의 외부 접속 주소를 localhost:9092~9094로 광고(KAFKA_ADVERTISED_LISTENERS)합니다.

    💡 왜 내부용(19092)과 외부용(9092) 포트를 따로 나누어야 하나요?

    카프카는 클라이언트(또는 다른 브로커)가 최초로 접속했을 때, "앞으로는 이 주소(KAFKA_ADVERTISED_LISTENERS)로 직접 찾아와!" 라고 자신의 통신 주소를 알려주는 특징이 있습니다. 이때 하나의 주소만 사용할 경우 도커 네트워크의 격리성 때문에 다음과 같은 모순이 발생합니다.

    1. 외부망 주소(localhost:9092)만 알려줄 경우: 내 PC(호스트)의 Java 앱은 접속에 성공합니다. 하지만 도커 내부의 다른 브로커(kafka-2)가 kafka-1을 찾기 위해 localhost:9092로 접속을 시도하면, 도커 안에서 localhost는 자기 자신을 의미하므로 엉뚱한 곳에서 헤매다 클러스터가 붕괴됩니다.
    2. 내부망 주소(kafka-1:9092)만 알려줄 경우: 브로커들끼리는 도커망을 통해 서로 잘 찾지만, 내 PC의 Java 앱은 kafka-1이라는 도커 내부의 호스트명을 인식하지 못해(DNS 찾기 실패) 접속에 실패합니다.
    3. 같은 포트에 두 개의 주소를 할당(INTERNAL://...:9092, EXTERNAL://...:9092)할 경우: 물리적으로 하나의 컨테이너 안에서 통신할 전체 개방 포트(0.0.0.0:9092)는 1개뿐이므로, Address already in use 에러가 발생하며 카프카가 실행되지 않습니다.

    결론적으로, 외부에서 접속할 포트(9092)와 브로커들끼리 도커망 안에서 조용히 대화할 내부용 포트(19092)를 분리하여 서로 길을 잃지 않도록 네트워크 교통정리를 하는 것입니다.

    C. 데이터 영속성 (Volumes)¶

    • docker-compose.yml의 volumes 설정(kafka-1-data 등)을 통해 컨테이너가 삭제되거나 재시작되어도 /var/lib/kafka/data 경로에 저장된 카프카 메시지와 클러스터 메타데이터가 내 컴퓨터 저장소에 안전하게 유지됩니다.

    3. Java 코드와의 연결 방식¶

    Java 애플리케이션 코드는 다음과 같이 설정되어 있습니다.

    private static final String BOOTSTRAP_SERVERS = "localhost:9092,localhost:9093,localhost:9094";

    카프카의 EXTERNAL 리스너 설정이 각 브로커를 localhost의 해당 포트들로 매핑하여 알려주므로, Java Producer는 로컬 환경에 설치된 카프카에 접속하는 것처럼 세 개의 브로커 모두에 분산 접속할 수 있습니다.

    4. 실행 및 토픽 생성 절차¶

    1단계: 클러스터 실행¶

    # 카프카 클러스터 백그라운드 실행
docker-compose up -d

    2단계: 토픽 생성 및 상태 확인¶

    주의: docker exec 명령어를 사용해 컨테이너 내부에서 스크립트를 실행할 때는 반드시 내부 통신용 포트(19092)를 사용해야 합니다. (외부용 9092 포트를 지정하면 메타데이터 에러가 발생합니다.)

    토픽 생성 (파티션 3개, 복제본 2개 구성)

    docker exec kafka-1 /opt/kafka/bin/kafka-topics.sh \
  --create --topic events \
  --bootstrap-server kafka-1:19092 \
  --partitions 3 \
  --replication-factor 2

    토픽 상태 확인

    docker exec kafka-1 /opt/kafka/bin/kafka-topics.sh \
  --describe --topic events \
  --bootstrap-server kafka-1:19092

    3단계: Java 애플리케이션 실행 및 동작 확인¶

    카프카 클러스터와 토픽 준비가 완료되면, 제공된 Java 애플리케이션을 통해 메시지 송수신 과정을 테스트합니다.

    A. 애플리케이션 빌드 및 실행 순서¶

    1. 빌드: 각 프로젝트 폴더(kafka-producer, kafka-consumer)에서 아래 명령어를 실행하여 실행 가능한 JAR 파일을 생성합니다. 
      mvn clean package
    2. Consumer 실행 (모니터링): 메시지 수신 상태를 실시간으로 확인하기 위해 Consumer를 먼저 실행합니다. 
      java -jar target/kafka.consumer-1.0-SNAPSHOT-jar-with-dependencies.jar [그룹ID]
    3. Producer 실행 (메시지 발행): Producer를 실행하여 10개의 메시지를 events 토픽으로 전송합니다. 
      java -jar target/kafka.producer-1.0-SNAPSHOT-jar-with-dependencies.jar

    B. 핵심 테스트 포인트¶

    1. 컨슈머 그룹 (Consumer Group) 활용 * 병렬 처리 (Scale-out): 동일한 그룹ID로 여러 개의 Consumer를 실행하면, 카프카는 토픽의 파티션(3개)을 컨슈머들에게 자동으로 분배합니다. (최대 3대까지 병렬 처리 가능) * 발행-구독 (Pub/Sub): 서로 다른 그룹ID를 가진 Consumer들은 동일한 메시지를 각각 독립적으로 수신합니다.

    2. 오프셋 관리 및 신뢰성 (Offset Commitment) * 본 예제는 enable.auto.commit=false로 설정되어 있으며, 메시지 처리 후 commitAsync()를 통해 명시적으로 오프셋을 기록합니다. * 테스트: Consumer가 메시지를 읽은 후 커밋하기 전에 강제 종료하면, 재시작 시 해당 메시지를 다시 읽어오는 '최소 한 번 전달(At least once)' 전략을 확인할 수 있습니다.

    3. 파티션 분산 저장 * Producer는 메시지 키(Key)를 사용하여 해시 기반으로 파티션을 결정합니다. 로그를 통해 10개의 메시지가 0~2번 파티션에 골고루 분산되어 저장되는지 확인하십시오.
