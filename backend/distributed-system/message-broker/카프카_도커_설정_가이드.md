# Kafka Docker Compose (KRaft 모드) 설정 가이드

이 문서는 프로젝트에 구성된 카프카 클러스터(`dhi.io/kafka:4`)의 설정 방식과 실행 방법, 그리고 Java 클라이언트와의 연결 구조를 설명합니다.

## 1. 개요

작성된 설정은 **KRaft(Kafka Raft)** 모드를 사용합니다. 이는 기존의 Zookeeper 없이 카프카 브로커들이 스스로 클러스터 상태를 관리하는 최신 방식입니다. 3개의 브로커 노드가 하나의 클러스터로 묶여 실행됩니다.

사용 중인 `dhi.io/kafka:4` (Hardened Kafka) 이미지는 일반적인 카프카 이미지들과 달리 환경 변수(`KAFKA_*`)를 통한 설정을 무시하고 내부의 `server.properties` 파일을 직접 읽어 실행됩니다. 따라서 별도의 셸 스크립트(`setup.sh`)를 통해 각 노드별 설정 파일을 생성한 후 컨테이너에 마운트하는 방식을 사용합니다.

## 2. 주요 설정 상세 설명

### A. 설정 파일

- **`node.id`**: 클러스터 내 브로커를 식별하는 고유 ID입니다. (1, 2, 3)
- **`process.roles=broker,controller`**: 각 노드가 데이터 저장(broker)과 클러스터 관리(controller) 역할을 모두 수행함을 의미합니다.
- **`controller.quorum.voters`**: 컨트롤러 역할을 하는 노드들의 목록으로, 브로커들이 서로를 인식하고 리더를 선출할 때 사용됩니다.

`docker-compose.yml` 파일의 `volumes` 설정을 통해 생성된 각 설정 파일이 컨테이너 내부의 `/opt/kafka/config/server.properties`로 덮어씌워집니다.

### B. 네트워크 및 리스너 분리 (Listeners)

클러스터 내부 통신과 외부(Java 클라이언트 등) 접근을 분리하여 포트 충돌 및 통신 오류를 방지합니다.

- **INTERNAL 망 (`19092` 포트)**: 도커 네트워크 내부에서 브로커들끼리 서로 통신할 때 사용됩니다. (예: `kafka-1:19092`)
- **EXTERNAL 망 (`9092` 포트)**: 호스트(사용자 PC)에서 접속할 때 사용됩니다. `docker-compose.yml`의 `ports` 설정을 통해 각각 `9092, 9093, 9094` 포트로 매핑되며, 브로커는 클라이언트에게 자신의 외부 접속 주소를 `localhost:9092~9094`로 광고(`advertised.listeners`)합니다.

**💡 왜 내부용(19092)과 외부용(9092) 포트를 따로 나누어야 하나요?**

카프카는 클라이언트(또는 다른 브로커)가 최초로 접속했을 때, **"앞으로는 이 주소(`advertised.listeners`)로 직접 찾아와!"** 라고 자신의 통신 주소를 알려주는 특징이 있습니다. 이때 하나의 주소만 사용할 경우 도커 네트워크의 격리성 때문에 다음과 같은 모순이 발생합니다.

1. **외부망 주소(`localhost:9092`)만 알려줄 경우**: 내 PC(호스트)의 Java 앱은 접속에 성공합니다. 하지만 도커 내부의 다른 브로커(`kafka-2`)가 `kafka-1`을 찾기 위해 `localhost:9092`로 접속을 시도하면, 도커 안에서 `localhost`는 **자기 자신**을 의미하므로 엉뚱한 곳에서 헤매다 클러스터가 붕괴됩니다.
2. **내부망 주소(`kafka-1:9092`)만 알려줄 경우**: 브로커들끼리는 도커망을 통해 서로 잘 찾지만, 내 PC의 Java 앱은 `kafka-1`이라는 도커 내부의 호스트명을 인식하지 못해(DNS 찾기 실패) 접속에 실패합니다.
3. **같은 포트에 두 개의 주소를 할당(`INTERNAL://...:9092, EXTERNAL://...:9092`)할 경우**: 물리적으로 하나의 컨테이너 안에서 통신할 전체 개방 포트(`0.0.0.0:9092`)는 1개뿐이므로, `Address already in use` 에러가 발생하며 카프카가 실행되지 않습니다.

**결론적으로, 외부에서 접속할 포트(`9092`)와 브로커들끼리 도커망 안에서 조용히 대화할 내부용 포트(`19092`)를 분리하여 서로 길을 잃지 않도록 네트워크 교통정리를 하는 것입니다.**

### C. 데이터 영속성 (Volumes)

- `docker-compose.yml`의 `volumes` 설정(`kafka-1-data` 등)을 통해 컨테이너가 삭제되거나 재시작되어도 `/opt/kafka/logs` 경로에 저장된 카프카 메시지와 클러스터 메타데이터가 내 컴퓨터 저장소에 안전하게 유지됩니다.

## 3. Java 코드와의 연결 방식

Java 애플리케이션 코드는 다음과 같이 설정되어 있습니다.

```java
private static final String BOOTSTRAP_SERVERS = "localhost:9092,localhost:9093,localhost:9094";
```

설정 파일의 EXTERNAL 리스너가 각 브로커를 `localhost`의 해당 포트들로 매핑하여 알려주므로, Java Producer는 로컬 환경에 설치된 카프카에 접속하는 것처럼 세 개의 브로커 모두에 자연스럽게 분산 접속할 수 있습니다.

## 4. 실행 및 토픽 생성 절차

### 1단계: 설정 파일 생성 및 클러스터 실행

```bash
# 1. 기존 데이터 초기화 (이전 클러스터 정보 충돌 방지)
docker-compose down -v

# 2. 카프카 클러스터 백그라운드 실행
docker-compose up -d
```

### 2단계: 토픽 생성 및 상태 확인

**주의:** `docker exec` 명령어를 사용해 컨테이너 내부에서 스크립트를 실행할 때는 반드시 **내부 통신용 포트(`19092`)**를 사용해야 합니다. (외부용 9092 포트를 지정하면 `The node does not support METADATA` 에러가 발생합니다.)

**토픽 생성 (파티션 3개, 복제본 2개 구성)**

```bash
docker exec kafka-1 /opt/kafka/bin/kafka-topics.sh \
  --create --topic events \
  --bootstrap-server kafka-1:19092 \
  --partitions 3 \
  --replication-factor 2
```

**토픽 상태 확인**

```bash
docker exec kafka-1 /opt/kafka/bin/kafka-topics.sh \
  --describe --topic events \
  --bootstrap-server kafka-1:19092
```

### 3단계: Java 애플리케이션 실행

카프카 클러스터와 토픽 준비가 완료되면, Java Producer 애플리케이션(`Application.java`)을 실행하여 메시지가 파티션에 분산되어 정상적으로 전송되는지 확인합니다.
