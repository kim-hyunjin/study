    지속 가능한 소프트웨어를 위한 설계: 디자인 패턴과 리팩토링¶

    본 프로젝트는 시간이 흘러도 유지보수가 용이하고 확장이 유연한 코드를 작성하기 위한 소프트웨어 설계 원칙과 디자인 패턴, 그리고 리팩토링 기법을 학습한 공간입니다.

    🏛 1. 객체지향 설계의 5가지 원칙 (SOLID)¶

    좋은 설계의 기초가 되는 SOLID 원칙을 실무 코드로 체득합니다. - SRP (Single Responsibility): 클래스는 하나의 책임만 가져야 한다. - OCP (Open/Closed): 확장에는 열려 있고 수정에는 닫혀 있어야 한다. - LSP (Liskov Substitution): 자식 클래스는 부모 클래스를 대체할 수 있어야 한다. - ISP (Interface Segregation): 사용하지 않는 메서드에 의존하지 않도록 인터페이스를 분리한다. - DIP (Dependency Inversion): 추상화에 의존해야 하며 구체화에 의존하면 안 된다.

    🛠 2. 주요 디자인 패턴 (Design Patterns)¶

    반복되는 문제들을 해결하기 위한 검증된 솔루션들을 학습합니다.

    생성 패턴 (Creational Patterns)¶

    • Singleton: 인스턴스를 하나만 생성하여 전역적으로 공유.
    • Factory Method: 객체 생성 로직을 자식 클래스나 별도 클래스로 분리.

    구조 패턴 (Structural Patterns)¶

    • Adapter: 서로 다른 인터페이스를 연결하여 호환성 확보.
    • Decorator: 객체의 코드를 수정하지 않고 동적으로 기능을 확장.

    행위 패턴 (Behavioral Patterns)¶

    • Observer: 객체의 상태 변화를 관찰자들에게 통지.
    • Strategy: 알고리즘을 캡슐화하여 상황에 따라 동적으로 교체.

    🧼 3. 리팩토링과 코드의 구린내 (Code Smell)¶

    작동하는 코드를 넘어, "깨끗한 코드"로 가꾸는 과정을 실습합니다.

    대표적인 코드의 구린내 (Smells)¶

    • 기나긴 함수 (Long Function): 한 함수가 너무 많은 일을 할 때.
    • 방대한 클래스 (Large Class): 인스턴스 변수가 너무 많거나 책임이 비대할 때.
    • 중복 코드 (Duplicated Code): 동일한 로직이 여러 곳에 흩어져 있을 때.

    해결 기법¶

    • 메서드 추출 (Extract Method): 긴 함수의 일부분을 별도의 함수로 분리.
    • 클래스 추출 (Extract Class): 비대해진 클래스의 책임을 나누어 새로운 클래스 생성.
    • 조건문 분해 (Decompose Conditional): 복잡한 조건식을 의미 있는 이름의 변수나 함수로 변환.

    📈 학습 포인트¶

    • Design vs Implementation: 구현 기술보다 중요한 설계의 본질 이해.
    • Refactoring with Tests: 테스트 코드가 뒷받침되지 않은 리팩토링은 위험하다는 사실 체감.
    • Patterns over Tools: 특정 라이브러리보다 범용적인 설계 패턴의 강력함 인식.

    본 저장소는 '더 나은 코드를 향한 끝없는 고민'의 흔적입니다.
