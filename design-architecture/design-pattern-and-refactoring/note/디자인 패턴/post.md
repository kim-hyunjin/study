# 디자인 패턴의 정석: 실무 적용 가이드

## 🚀 개요
소프트웨어 설계에서 반복적으로 발생하는 문제들에 대한 검증된 해결책인 **디자인 패턴**을 정리합니다. 단순한 이론을 넘어, 각 패턴이 해결하고자 하는 핵심 문제와 패턴 간의 미묘한 차이점을 중심으로 살펴봅니다.

## 💡 객체지향 설계의 황금률
패턴을 배우기 전, 모든 디자인 패턴의 근간이 되는 원칙을 기억해야 합니다.
- **캡슐화:** 바뀌는 부분은 따로 뽑아서 캡슐화한다.
- **구성(Composition):** 상속보다는 구성을 활용한다.
- **인터페이스 지향:** 구현이 아닌 인터페이스에 맞춰서 프로그래밍한다.
- **느슨한 결합:** 상호작용하는 객체 사이에는 느슨한 결합을 유지한다.

## 🛠 주요 디자인 패턴 분석

### 1. 전략 패턴 (Strategy Pattern)
알고리즘군을 정의하고 각각을 캡슐화하여 실행 중에 교체 가능하게 만듭니다.
- **핵심:** 클라이언트와 독립적으로 알고리즘을 변경할 수 있습니다.
- **예시:** 오리의 나는 행동, 꽥꽥거리는 행동을 인터페이스로 분리하여 동적으로 변경.

### 2. 데코레이터 패턴 (Decorator Pattern)
객체에 추가 요소를 동적으로 더하여 기능을 확장합니다.
- **장점:** 서브클래스를 만드는 것보다 훨씬 유연하게 기능을 확장할 수 있습니다.
- **특징:** 데코레이터는 자신이 감싸고 있는 구성요소와 같은 형식을 가집니다.

### 3. 스테이트 vs 스트래티지 패턴
두 패턴은 구조적으로 비슷하지만 **용도**가 다릅니다.
- **스테이트 패턴:** 객체의 **내부 상태**에 따라 행동이 바뀌며, 상태 전환을 캡슐화합니다.
- **스트래티지 패턴:** 클라이언트가 **어떤 전략**을 사용할지 직접 지정하며, 주로 상속의 대안으로 쓰입니다.

## 📐 아키텍처 비교: 래퍼(Wrapper) 패턴들
비슷해 보이는 패턴들을 목적에 따라 구분하는 것이 중요합니다.

| 패턴 | 목적 |
| :--- | :--- |
| **데코레이터** | 객체를 감싸서 **새로운 행동**을 추가 |
| **어댑터** | 객체를 감싸서 **인터페이스**를 변환 |
| **퍼사드** | 일련의 객체들을 감싸서 **인터페이스를 단순화** |

## 📝 배운 점 및 결론
- **오남용 주의:** 디자인 패턴은 목적이 아니라 수단입니다. 가장 간단한 해결책이 최선일 때가 많습니다.
- **유연성의 비용:** 패턴을 적용하면 클래스 개수가 늘어나고 복잡도가 증가할 수 있습니다. 이 비용이 유연성이라는 이득보다 큰지 항상 고민해야 합니다.
- **패턴은 도구일 뿐:** 필요에 따라 변형해서 사용하는 유연함이 필요합니다.
