    코드의 구린내(Code Smell)와 리팩토링 전략¶

    🚀 개요¶

    리팩토링은 소프트웨어의 겉보기 동작은 그대로 유지하면서, 내부 구조를 개선하여 코드를 이해하기 쉽고 수정하기 용이하게 만드는 과정입니다. 리팩토링을 언제 시작해야 할까요? 바로 코드에서 '구린내(Smell)'가 날 때입니다.

    💡 주요 코드 구린내와 해결책¶

    1. 중복 코드 (Duplicated Code)¶

    "구린내의 제왕"입니다. 똑같은 코드 구조가 두 군데 이상 있다면 반드시 하나로 합쳐야 합니다. - 해결책: 메서드 추출(Extract Method), 메서드 상향(Pull Up Method), 알고리즘 전환 등

    2. 장황한 메서드 (Long Method)¶

    객체지향 프로그램의 강점은 짧은 메서드에서 나옵니다. 메서드가 길어지면 의도를 파악하기 어렵고 오버헤드가 발생합니다. - 핵심: 주석을 달아야 할 것 같은 부분은 과감히 메서드로 추출하고, 의도가 잘 드러나는 이름을 붙입니다. - 해결책: 메서드 추출, 임시 변수를 메서드 호출로 전환 등

    3. switch 문¶

    객체지향 코드에서 switch 문은 다형성(Polymorphism)으로 대체되어야 할 대상입니다. - 문제점: 같은 switch 문이 여러 곳에 중복될 가능성이 높고, 새로운 조건을 추가할 때마다 모든 곳을 수정해야 합니다. - 해결책: 조건문을 재정의(Polymorphism)로 전환.

    🛠 리팩토링 결정 트리 (요약)¶

    구린내 증상 처방
    수정의 산발 한 클래스가 여러 이유로 자주 바뀜 클래스 추출
    기능의 산재 한 수정사항을 위해 여러 클래스를 고침 메서드/필드 이동
    잘못된 소속 메서드가 남의 클래스 데이터를 더 많이 씀 메서드 이동
    데이터 뭉치 여러 데이터가 항상 몰려다님 클래스 추출

    📐 아키텍처 개선 기법: 위임과 상속¶

    클래스 간의 관계가 너무 밀접하거나(지나친 관여), 단순히 기능을 전달만 하는 경우(과잉 중개 메서드) 다음과 같은 전략을 사용합니다. - 대리 객체 은폐: 클라이언트가 몰라도 되는 중간 객체는 숨깁니다. - 상속을 위임으로 전환: 하위 클래스가 상위 클래스의 인터페이스를 거부하거나 너무 많은 데이터에 관여할 때 사용합니다.

    📝 배운 점 및 결론¶

    • 주석은 탈취제다: 코드가 복잡해서 주석이 필요하다면, 주석을 쓰기 전에 리팩토링을 통해 코드 자체의 가독성을 높여야 합니다.
    • 점진적 개선: 리팩토링은 한꺼번에 하는 것이 아니라, 기능을 추가하거나 버그를 잡을 때마다 조금씩 수행하는 습관이 중요합니다.
    • 테스트의 중요성: 리팩토링의 전제 조건은 견고한 테스트 코드입니다. 구조를 바꿔도 동작이 변하지 않았음을 보장받아야 합니다.
